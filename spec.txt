Especificação inicial — Robotrace Sim v0.08
1. Objetivo do projeto

Criar um simulador de robôs seguidores de linha com foco em realismo físico, modularidade e reaproveitamento parcial dos dados em robôs reais.

O simulador deve permitir:

1. Criar e editar pistas.
2. Criar e editar robôs.
3. Simular o comportamento físico do robô na pista.
4. Testar controladores.
5. Executar simulações por interface gráfica.
6. Executar simulações por linha de comando.
7. Exportar logs, replays e dados para análise.
8. Comparar simulação com dados reais.

A simulação deve trabalhar com fixed-step determinístico, usando tempo interno em ticks inteiros de microssegundos.

Configuração base:

physics_dt_us = 500
physics_rate = 2000 Hz

Ou seja:

0,5 ms por passo
2000 passos físicos para cada 1 segundo simulado
2. Decisões técnicas principais

Ficam definidas como base:

Item	Decisão
Linguagem principal	Rust
Interface	egui/eframe
Renderização inicial	egui painter
Renderização futura	wgpu, se necessário
Formato do projeto	.rtsim, com JSON versionado
Física	própria, determinística, fixed-step
Tempo interno	ticks inteiros de microssegundos
Controlador	Rust built-in + plugin nativo
Logs	replay binário + CSV exportável
Testes	unitários, benchmarks e comparação com dados reais
Execução automatizada	sim, via comando de linha
3. Arquitetura geral

O projeto deve ter um único executável:

robotrace-sim.exe

Esse executável deve conter:

Home
Editor de pista
Editor de robô
Editor de simulação
Simulador
Replay viewer
Ferramenta de calibração
Ferramenta de benchmark

Mas internamente, o projeto deve ser dividido em módulos independentes.

Estrutura conceitual:

App/UI
 ├── Track Editor
 ├── Robot Editor
 ├── Simulation Editor
 ├── Simulator Viewer
 ├── Replay Viewer
 └── Calibration Tools

Simulation Core
 ├── Scheduler
 ├── Physics Engine
 ├── Robot Model
 ├── Track Model
 ├── Sensor Models
 ├── Controller API
 ├── Telemetry
 └── Replay System

A interface gráfica não deve controlar a física diretamente. A UI apenas configura, visualiza e envia comandos. O núcleo de simulação deve funcionar sozinho, inclusive por terminal.

4. Execução por linha de comando

Esse requisito é importante e deve existir desde o começo, não como recurso futuro.

Exemplos desejados:

robotrace-sim run projeto.rtsim
robotrace-sim run projeto.rtsim --headless
robotrace-sim run projeto.rtsim --duration 10s --output resultado.rtlog
robotrace-sim batch ./experimentos/*.rtsim --output ./resultados/
robotrace-sim benchmark projeto.rtsim --physics-dt-us 500
robotrace-sim export resultado.rtlog --format csv

Isso permite testar vários robôs, controladores e parâmetros automaticamente.

5. Conceito central: simulação modular

Cada parte do robô e do ambiente deve ser um módulo selecionável. A ideia é que o usuário possa escolher:

modelo simples
modelo intermediário
modelo avançado
modelo experimental/customizado

Por exemplo, para pneu:

IdealTire
BasicFrictionTire
SlipRatioTire
LoadSensitiveTire
ThermalWearTire

Para motor:

IdealMotor
DcMotorSimple
DcMotorElectrical
CorelessDcMotor
BrushlessMotor
MeasuredMotorCurve

Para sensor:

IdealLineSensor
ReflectanceSensor
AnalogIrSensor
OpticalArraySensor
MeasuredSensorModel

Assim, o simulador fica maleável: pode rodar rápido com modelos simples ou rodar mais realista com modelos avançados.

6. Módulos principais da simulação
6.1. Módulo de tempo e scheduler

Esse módulo define quando cada sistema roda.

Nem tudo deve rodar a 2000 Hz. A física pode rodar a 2000 Hz, mas sensores, controle e logs podem ter frequências próprias.

Exemplo:

{
  "time": {
    "physics_dt_us": 500,
    "controller_period_us": 1000,
    "sensor_period_us": 500,
    "imu_period_us": 500,
    "encoder_period_us": 500,
    "log_period_us": 1000,
    "render_period_us": 16667
  }
}

O scheduler deve garantir:

Física: fixed-step
Controle: período configurável
Sensores: período configurável
Renderização: desacoplada
Log: desacoplado
6.2. Módulo de pista

A pista não deve ser apenas uma imagem. Ela deve ter uma representação vetorial e uma representação rasterizada.

Camadas da pista:

Track Geometry
 ├── retas
 ├── curvas
 ├── arcos
 ├── splines
 ├── cruzamentos
 ├── marcações
 └── largada/chegada

Track Surface
 ├── material da base
 ├── material da linha
 ├── coeficiente de atrito
 ├── refletância
 ├── irregularidade
 ├── sujeira
 └── emendas

Modelos selecionáveis:

Módulo	Função
IdealTrack	pista perfeita, linha ideal
VectorTrack	pista vetorial com largura real da linha
RasterReflectanceTrack	mapa raster de refletância para sensores
SurfaceFrictionTrack	atrito variável por região
DirtyTrack	ruído espacial, sujeira e falhas
BumpyTrack	pequenas irregularidades de superfície

A pista deve fornecer duas informações principais:

1. Refletância no ponto/área consultada.
2. Propriedades de contato pneu-solo.
6.3. Módulo de corpo/chassi

O robô deve ter massa, centro de massa, momento de inércia e geometria.

Parâmetros:

massa total
posição do centro de massa
momento de inércia em yaw
comprimento
largura
altura
posição das rodas
posição dos sensores
posição da bateria
posição dos fans/sucção
posição do centro aerodinâmico/downforce

Modelos:

Módulo	Função
PointMassChassis	modelo simples, massa pontual
RigidBody2DChassis	corpo rígido 2D com inércia
LoadTransferChassis	considera transferência dinâmica de carga
FlexibleMountChassis	modelo futuro para vibração/sensor/fan

Para começar, eu usaria RigidBody2DChassis.

6.4. Módulo de rodas e pneus

Esse é um dos módulos mais importantes para realismo.

O robô pode ter:

2 rodas
4 rodas
rodas auxiliares
rodas passivas
rodas de apoio

Parâmetros por roda:

posição no chassi
raio
largura
massa/inércia
material
coeficiente de atrito longitudinal
coeficiente de atrito lateral
coeficiente de rolamento
limite de força
modelo de slip

Modelos selecionáveis:

Módulo	Realismo	Descrição
IdealWheel	baixo	sem deslizamento
CoulombFrictionWheel	médio	limite simples por μN
SlipRatioWheel	alto	considera diferença entre velocidade da roda e do solo
CombinedSlipWheel	alto	combina slip longitudinal e lateral
LoadSensitiveWheel	alto	atrito depende da carga normal
MeasuredTireModel	muito alto	curva calibrada com dados reais

O simulador deve conseguir representar o caso em que o motor aplica torque demais, a roda começa a patinar e o robô acelera menos do que o esperado.

Relação física essencial:

F_max = μ * N

Onde:

F_max = força máxima transmissível pelo pneu
μ = coeficiente de atrito
N = força normal na roda

Com fan ou sucção:

N_total = peso + força_extra_de_aderência

Isso é obrigatório para simular robôs como os das imagens anexadas.

6.5. Módulo de normal/aderência/downforce

Esse módulo merece existir separado, porque afeta diretamente o limite de tração, frenagem e curva.

O usuário citou três casos:

1. Nenhum sistema de aumento de normal.
2. Sistema de sucção.
3. Sistema de ventoinha/downforce.

Então eu criaria um módulo chamado:

NormalForceModule

Modelos selecionáveis:

Módulo	Descrição
NoDownforce	apenas peso do robô
ConstantDownforce	força extra constante
FanDownforce	downforce gerado por uma ou mais ventoinhas
SuctionDownforce	força por pressão negativa sob o robô
SpeedDependentAero	força varia com velocidade do ar
MeasuredDownforceCurve	curva real medida
ClosedLoopDownforce	fan/sucção controlado por PWM
Caso 1 — sem sistema
N_total = m * g

Distribuição nas rodas depende do centro de massa.

Caso 2 — sucção

Inspirado no RT61 APOLLON, que usa um mecanismo de sucção com motor DC dedicado e descreve uma estrutura de absorção de vibração no fan de sucção.

Modelo simples:

F_suction = constante

Modelo intermediário:

F_suction = f(pwm, rpm_fan, vazamento, altura_do_selo, velocidade)

Modelo avançado:

pressão interna da câmara
vazamento lateral
área efetiva de sucção
tempo de resposta do fan
perda por irregularidade da pista
perda por inclinação/chassi

Equação base:

F_suction = ΔP * A_effective

Onde:

ΔP = diferença de pressão
A_effective = área efetiva da câmara de sucção
Caso 3 — ventoinha/downforce

Inspirado no Aegis2023, que usa 4 motores DC BetaFPV como downforce fans.

Modelo simples:

F_fan = constante

Modelo intermediário:

F_fan = k * rpm²

Modelo avançado:

rpm do fan
PWM do fan
tensão da bateria
corrente do fan
curva empuxo x rpm
posição dos fans no chassi
momento gerado pelo downforce
interferência com altura em relação ao solo

Esse módulo deve devolver não só força vertical total, mas também distribuição de carga:

normal_front_left
normal_front_right
normal_rear_left
normal_rear_right

Isso é importante porque um fan colocado mais à frente aumenta a normal nas rodas dianteiras ou pode alterar o comportamento em curva.

O simulador deve suportar vários tipos de motor.

Modelos:

Módulo	Uso

DcMotorSimple	motor DC com torque proporcional
DcMotorElectrical	considera R, L, Ke, Kt
CorelessDcMotor	baixa inércia, resposta rápida

BrushlessMotorSimple	BLDC simplificado
BrushlessMotorFOC	BLDC com controle de corrente/FOC
MeasuredMotorCurve	curva real medida em bancada

Parâmetros para motor DC:

indutância

constante de força contraeletromotriz Ke
corrente sem carga
corrente de stall
rotação sem carga
torque de stall
inércia do rotor
atrito interno

Equações básicas:

V = R*i + L*di/dt + Ke*ω
T = Kt*i

O modelo avançado deve permitir saturação e limite térmico.

Parâmetros por sensor:

ruído
saturação

Pipeline recomendado:

→ refletância média

→ filtro
→ quantização ADC

Para os robôs de referência, isso é coerente porque ambos usam 16 sensores infravermelhos de linha.

7. Estrutura inicial dos módulos no Rust

A ideia em Rust seria criar traits para cada família de módulo.

Exemplo conceitual:

pub trait MotorModel {
    fn step(&mut self, input: MotorInput, dt_us: u32) -> MotorOutput;
}

pub trait TireModel {
    fn compute_forces(&mut self, input: TireInput) -> TireForces;
}

pub trait SensorModel {
    fn sample(&mut self, input: SensorInput, t_us: u64) -> SensorOutput;
}

pub trait NormalForceModel {
    fn compute_normal_forces(&mut self, input: NormalForceInput) -> NormalForceOutput;
}

pub trait BatteryModel {
    fn step(&mut self, input: BatteryInput, dt_us: u32) -> BatteryOutput;
}

O importante é que cada módulo tenha interface própria e possa ser trocado por configuração.
